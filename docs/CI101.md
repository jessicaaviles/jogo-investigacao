CI 101

O que é CI (Integração Contínua)
- CI é a prática de automatizar a construção, testes e validações do código sempre que há mudanças no repositório, para detectar erros cedo e manter o código estável.

O que este repositório faz com CI
- Há um workflow de GitHub Actions chamado Build and Test que executa em pushes e pull requests.
- Frontend: instala dependências, constrói o frontend com Vite, gerando a bundle de produção.
- Backend: instala dependências, compila o TypeScript e executa os testes do backend.
- Logs ficam disponíveis na aba Actions para cada execução, com detalhes dos passos e falhas, se houver.

Como ver o status do CI
- No GitHub, vá para a aba Actions; selecione o workflow Build and Test para ver as execuções recentes e os logs detalhados.
- Em caso de falha, abra a execução correspondente e leia os logs para entender a origem do erro.

Como rodar localmente os passos equivalentes ao CI
- Frontend
  1) Navegue até a pasta frontend.
  2) Execute: npm ci
  3) Execute: npm run build
- Backend
  1) Navegue até a pasta backend.
  2) Execute: npm ci
  3) Execute: npm run build
  4) Execute: npm test

Notas de configuração local
- Não versionamos segredos. Use backend/.env.local ou backend/.env e não commit seus valores.
- Para replicar o ambiente de CI o mais fiel possível, use Node.js na versão suportada pelo workflow (geralmente a versão especificada no workflow).

Problemas comuns e soluções rápidas
- Erros de permissão ao instalar dependências: feche antivírus/softwares de segurança ou rode com permissões elevadas conforme o seu OS permite.
- Erros de rede ao npm ci: verifique conectividade, proxy e caches.
- Diferenças de ambiente entre Windows e Linux: alguns comandos podem ter pequenas diferenças; prefira rodar o CI no ambiente Linux (GitHub Actions) para consistência.

Resumo

- Observação: este guia está focado em CI para equipes novas. Se já estiver familiar, utilize os passos de verificação locais conforme necessário.
- A CI ajuda a detectar problemas cedo, mantendo o repositório estável. Se algo falhar, leia os logs da run de CI e aplique correções no código, depois reenvie as mudanças (push/PR).
